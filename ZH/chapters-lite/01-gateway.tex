\chapter{Gateway（网关核心）}

\noindent\textit{上一章我们从宏观视角了解了 OpenClaw 的整体架构和五大核心组件。本章将深入第一个核心组件——Gateway 网关，
它是 OpenClaw 的“中枢神经系统”，负责消息路由、会话管理和智能体运行时调度。
理解 Gateway 是掌握 OpenClaw 工作流程的关键。}

\section{概述}
%----------------------------------------------------------------------

Gateway 网关是 OpenClaw 的\textbf{核心进程}，是整个系统的"中央控制塔"。
正如机场需要空中管制中心来调度每一架航班的起降，Gateway 负责协调所有消息的流入、处理和回传。
它以单进程方式运行，提供统一的 WebSocket + HTTP 多路复用端口，是 OpenClaw 中唯一拥有会话状态权威性的组件。

Gateway 承担六大核心职责：

\begin{enumerate}
  \item \textbf{消息路由}：接收各渠道入站消息，根据 Binding 规则路由到正确的 Agent
  \item \textbf{会话管理}：维护所有会话状态（唯一事实来源），包括上下文、Token 计数、压缩历史
  \item \textbf{智能体运行时}：内嵌 pi-agent-core，执行完整的智能体推理--执行循环
  \item \textbf{RPC 服务}：对外提供 WebSocket + HTTP 双协议 API，支持 CLI、Web UI、移动节点接入
  \item \textbf{插件宿主}：加载并运行所有已注册插件，扩展 Gateway 的能力边界
  \item \textbf{定时任务}：内置 Cron 调度器，支持定时唤醒、周期性检查等自动化场景
\end{enumerate}

\noindent 配置路径：\texttt{gateway.*}（位于 \texttt{\textasciitilde{}/.openclaw/openclaw.json}）

\section{启动方式}
%----------------------------------------------------------------------

Gateway 支持两种主要的启动方式：前台运行（开发/调试）和守护进程模式（生产部署）。

\subsection{前台运行}

直接在终端启动 Gateway，适合开发和调试：

\begin{lstlisting}[language=bash]
openclaw gateway --port 18789
\end{lstlisting}

常用的启动选项：

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{选项} & \textbf{默认值} & \textbf{说明} \\
\midrule
\texttt{--port} & \texttt{18789} & WebSocket + HTTP 多路复用端口 \\
\texttt{--dev} & --- & 开发模式，启用额外的调试日志 \\
\texttt{--verbose} & --- & 详细日志输出 \\
\texttt{--compact} & --- & 紧凑的 WebSocket 日志样式 \\
\texttt{--force} & --- & 强制启动，杀死端口上已有的监听器 \\
\bottomrule
\end{tabular}
\end{center}

\noindent 端口优先级：\texttt{--port} 参数 $>$ \texttt{OPENCLAW\_GATEWAY\_PORT} 环境变量 $>$ 配置文件 \texttt{gateway.port} $>$ 默认值 \texttt{18789}。

\subsection{守护进程模式（推荐生产使用）}

将 Gateway 安装为操作系统的守护进程服务，实现开机自启和异常自动重启：

\begin{lstlisting}[language=bash]
openclaw gateway install   # Install systemd/launchd/schtasks service
openclaw gateway start     # Start the daemon
openclaw gateway status    # Check status
openclaw gateway restart   # Restart the daemon
openclaw gateway stop      # Stop the daemon
openclaw gateway uninstall # Remove the service
\end{lstlisting}

\section{智能体循环（Agent Loop）}
%----------------------------------------------------------------------

Gateway 的核心工作是处理\textbf{智能体循环}——每条入站消息都会触发一次完整的推理--执行循环。
这是 OpenClaw 区别于简单聊天机器人的关键所在：它不仅能回复文本，还能调用工具并执行实际操作。

\begin{center}

    toolstep/.style={draw=orange!60, fill=orange!8, rounded corners=4pt,
        minimum width=12.5cm, minimum height=0.8cm, font=\small, align=left},
    arr/.style={-{Stealth[length=2.5mm]}, thick, draw=teal!60},
    arrt/.style={-{Stealth[length=2.5mm]}, thick, draw=orange!60}
]
    \node[agentstep] (s1) {\quad 1. \textbf{接收消息}：验证来源、解析会话键、检查速率限制};
    \node[agentstep, below=of s1] (s2) {\quad 2. \textbf{路由决策}：根据 Binding 规则匹配目标 Agent};
    \node[agentstep, below=of s2] (s3) {\quad 3. \textbf{加载上下文}：读取会话历史、Skills 快照、Memory 检索结果};
    \node[agentstep, below=of s3] (s4) {\quad 4. \textbf{组装提示}：基础提示 + Skills 提示 + 引导上下文 + 用户消息};

\end{tikzpicture}
\end{center}

\noindent \textbf{重要特性}：

\begin{itemize}
  \item 步骤 5--7 构成\textbf{工具调用循环}：模型可以连续调用多个工具，直到得出最终回复
  \item 流式输出：模型回复以流式方式实时推送给客户端，用户无需等待完整回复
  \item 并发控制：同一会话内的多条消息会按序处理，避免上下文冲突
  \item 工具调用支持在 Docker 沙箱中执行高风险操作（如 Shell 命令）
\end{itemize}

\noindent \textbf{关键点}：

\begin{itemize}
  \item 第一帧\textbf{必须}是 \texttt{connect} 请求，否则连接被拒绝
  \item \texttt{hello-ok} 响应携带 snapshot（presence 列表、健康状态、协议版本等）
  \item \texttt{agent} 方法采用\textbf{两阶段响应}：首先返回 \texttt{accepted} 确认，流式推送中间结果，最终返回完整结果
  \item Gateway 定期发送 \texttt{tick} 事件作为心跳，客户端可据此检测连接活性
\end{itemize}

\section{RPC 接口}
%----------------------------------------------------------------------

Gateway 通过 WebSocket 和 HTTP 暴露丰富的 RPC 接口，涵盖从健康检查到智能体调用的各个方面：

\begin{center}
\small
\begin{tabular}{llp{5.5cm}}
\toprule
\textbf{类别} & \textbf{方法} & \textbf{说明} \\
\midrule
核心 & \texttt{connect} & 必须是第一帧，完成握手 \\
     & \texttt{health} & 完整健康快照 \\
     & \texttt{status} & 简短状态摘要 \\
\midrule
智能体 & \texttt{agent} / \texttt{agent.wait} & 触发智能体运行（流式/等待模式） \\

\midrule

     & \texttt{sessions.patch} & 修改会话属性 \\

     & \texttt{chat.history} & 获取聊天历史 \\
\midrule
消息 & \texttt{send} & 通过活跃渠道发送消息 \\
     & \texttt{poll} & 轮询渠道消息 \\
     & \texttt{message.send} & 发送消息到指定渠道 \\
\midrule
节点 & \texttt{node.list} & 列出已配对 + 在线节点 \\
     & \texttt{node.invoke} & 调用节点上的命令 \\
     & \texttt{node.pair.*} & 节点配对管理 \\
\midrule
系统 & \texttt{system-presence} & 当前在线列表 \\
     & \texttt{logs.tail} & 实时日志尾随 \\
\bottomrule
\end{tabular}
\end{center}

\noindent 通过 CLI 调用 RPC 接口：

\begin{lstlisting}[language=bash]
# List sessions
openclaw gateway call sessions.list --params '{}'

# Check health
openclaw gateway call health

# Tail logs
openclaw gateway call logs.tail --params '{"sinceMs": 60000}'
\end{lstlisting}

\noindent 有副作用的方法（如 \texttt{send}、\texttt{agent}）需要\textbf{幂等键}（\texttt{idempotencyKey}），
Gateway 维护短期去重缓存以保证安全重试。

\texttt{per-peer} & 跨渠道按发送者 ID 隔离 \\
\texttt{per-channel-peer} & 按渠道 + 发送者隔离（推荐多用户场景） \\
\texttt{per-account-channel-peer} & 按账户 + 渠道 + 发送者隔离（多账户收件箱） \\
\bottomrule
\end{tabular}
\end{center}

\subsection{会话存储}

每个会话在 \texttt{sessions.json} 中维护一条记录，关键字段包括：

\begin{itemize}
  \item \texttt{sessionId}：唯一标识符
  \item \texttt{chatType}：\texttt{direct | group | room}
  \item Token 计数器：\texttt{inputTokens}、\texttt{outputTokens}、\texttt{totalTokens}、\texttt{contextTokens}
  \item \texttt{compactionCount}：自动压缩次数
  \item 模型选择：支持按会话覆盖提供商、模型和认证配置
\end{itemize}

\noindent 对话记录存储为 JSONL 格式（每行一条消息），路径由 \texttt{sessionId} 派生。

\section{配置详解}
%----------------------------------------------------------------------

Gateway 的完整配置位于 \texttt{\textasciitilde{}/.openclaw/openclaw.json} 的 \texttt{gateway} 节点下。
以下是一个典型配置示例：

\begin{lstlisting}[language=json]
{
  gateway: {
    mode: "local",
    port: 18789,
    bind: "loopback",
    auth: {
      mode: "token",
      token: "your-secret-token",
      rateLimit: {
        maxAttempts: 10,
        windowMs: 60000,
        lockoutMs: 300000,
        exemptLoopback: true
      }
    },
    controlUi: {
      enabled: true,
      basePath: "/openclaw"
    },
    tailscale: {
      mode: "off"
    }
  }
}
\end{lstlisting}

\noindent 关键配置说明：

\begin{center}
\small
\begin{tabular}{llp{5cm}}
\toprule
\textbf{字段} & \textbf{可选值} & \textbf{说明} \\
\midrule
\texttt{mode} & \texttt{local} / \texttt{remote} & \texttt{local} 运行 Gateway，\texttt{remote} 连接远程 Gateway \\
\texttt{bind} & \texttt{loopback / lan / tailnet / custom} & 网络绑定模式，默认仅本地回环 \\

\end{tabular}

\subsection{配置热重载}

Gateway 监视配置文件变更并自动应用：

\begin{itemize}
  \item \textbf{hybrid}（默认）：安全更改热应用，关键更改触发重启
  \item \textbf{full}：所有更改都触发重启
  \item \textbf{off}：禁用热重载
\end{itemize}

\begin{lstlisting}[language=json]
{

    mode: "remote",
    remote: {
      url: "ws://gateway.tailnet:18789",
      transport: "ssh",
      token: "your-token"
    }
  }
}
\end{lstlisting}

\section{认证}
%----------------------------------------------------------------------

Gateway 支持多种认证方式来保护 API 访问和模型调用。

\subsection{Gateway 访问认证}

控制谁可以连接到 Gateway 的 WebSocket 端口：

\begin{itemize}
  \item \textbf{Token 认证}（推荐）：在 \texttt{gateway.auth.token} 中设置密钥，客户端在 \texttt{connect} 时提供
  \item \textbf{密码认证}：使用 \texttt{gateway.auth.password} 或环境变量 \texttt{OPENCLAW\_GATEWAY\_PASSWORD}
  \item \textbf{无认证}：仅适用于完全信任的本地环境
  \item \textbf{速率限制}：内置暴力破解防护，默认 60 秒内最多 10 次尝试
\end{itemize}

\subsection{模型认证}

推荐使用 \textbf{Anthropic API 密钥}：

\begin{lstlisting}[language=bash]
# Set API key as environment variable
export ANTHROPIC_API_KEY="..."

# Or store in daemon env file for persistence
echo 'ANTHROPIC_API_KEY=...' >> ~/.openclaw/.env

# Verify authentication status
openclaw models status
\end{lstlisting}

\noindent 也支持 \texttt{claude setup-token} 的订阅认证流程和 OAuth 模式。

\subsection{设备配对}

Gateway 使用基于设备的配对机制管理节点访问：

\begin{itemize}
  \item 新设备首次连接时需要配对审批
  \item 本地连接（loopback）可自动批准
  \item 审批后颁发设备令牌，存储在 \texttt{paired.json} 中
  \item 令牌轮换需要重新审批
\end{itemize}

%----------------------------------------------------------------------
\section{Gateway 锁与多实例}
%----------------------------------------------------------------------

Gateway 采用\textbf{独占 TCP 监听器}机制确保同一端口只运行一个实例：

\begin{itemize}
  \item 启动时立即绑定 WebSocket 端口，如果端口被占用则快速失败
  \item 操作系统在进程退出（包括崩溃和 SIGKILL）时自动释放监听器
  \item 无需额外的锁文件或清理步骤
\end{itemize}

\noindent 如果需要在同一台机器上运行多个 Gateway 实例（例如救援机器人），
需使用隔离的配置文件和唯一端口：

\begin{lstlisting}[language=bash]
OPENCLAW_CONFIG_PATH=~/.openclaw/rescue.json \
OPENCLAW_STATE_DIR=~/.openclaw-rescue \
  openclaw gateway --port 18790
\end{lstlisting}

\section{本章小结}
%----------------------------------------------------------------------

本章详细介绍了 Gateway 网关的核心职责和运作机制：

\begin{itemize}
  \item Gateway 是 OpenClaw 的\textbf{中央调度中心}，承担消息路由、会话管理、智能体运行时、RPC 服务、插件宿主和定时任务六大核心职责
  \item 每条入站消息触发一次完整的\textbf{智能体循环}（Agent Loop），支持多轮工具调用
  \item Gateway 使用自定义的 \textbf{WebSocket 协议}，采用 Request/Response/Event 三帧模型通信
  \item 会话管理遵循\textbf{唯一事实来源}原则，支持灵活的路由规则和隔离策略
  \item 支持\textbf{前台运行}和\textbf{守护进程}两种模式，以及跨平台的服务管理
  \item 内置丰富的\textbf{健康检查}和诊断工具，确保系统稳定运行
\end{itemize}

\noindent Gateway 已经跑起来了，但它调度的"大脑"是谁？下一章我们将深入了解 \textbf{Agent（智能体）}——OpenClaw 的思考和执行单元，了解它如何与 LLM 交互并完成复杂任务。
