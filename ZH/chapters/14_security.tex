\section{形式化验证（安全模型）}


本页跟踪 OpenClaw 的\textbf{形式化安全模型}（目前是 TLA+/TLC；根据需要会添加更多）。

\begin{quote}
注意：一些较旧的链接可能引用了以前的项目名称。
\end{quote}


\textbf{目标（北极星）：} 提供机器检查的论证，证明 OpenClaw 在明确假设下执行其
预期的安全策略（授权、会话隔离、工具门控和
配置错误安全）。

\textbf{目前是什么：} 一个可执行的、攻击者驱动的\textbf{安全回归测试套件}：

\begin{itemize}
  \item 每个声明都有一个在有限状态空间上运行的模型检查。
  \item 许多声明有一个配对的\textbf{负面模型}，为现实的 bug 类别生成反例追踪。
\end{itemize}


\textbf{目前还不是什么：} 证明"OpenClaw 在所有方面都是安全的"或完整 TypeScript 实现是正确的。

\subsection{模型存放位置}


模型维护在一个单独的仓库中：\href{https://github.com/vignesh07/openclaw-formal-models}{vignesh07/openclaw-formal-models}。

\subsection{重要注意事项}


\begin{itemize}
  \item 这些是\textbf{模型}，不是完整的 TypeScript 实现。模型和代码之间可能存在偏差。
  \item 结果受 TLC 探索的状态空间限制；"绿色"并不意味着在建模的假设和边界之外也是安全的。
  \item 一些声明依赖于明确的环境假设（例如，正确的部署、正确的配置输入）。
\end{itemize}


\subsection{复现结果}


目前，结果通过在本地克隆模型仓库并运行 TLC 来复现（见下文）。未来的迭代可能提供：

\begin{itemize}
  \item 带有公开产物（反例追踪、运行日志）的 CI 运行模型
  \item 用于小型、有界检查的托管"运行此模型"工作流
\end{itemize}


开始使用：

\begin{lstlisting}[language=bash]
git clone https://github.com/vignesh07/openclaw-formal-models
cd openclaw-formal-models

# 需要 Java 11+（TLC 在 JVM 上运行）。
# 仓库内置了固定版本的 `tla2tools.jar`（TLA+ 工具）并提供 `bin/tlc` + Make 目标。

make <target>
\end{lstlisting}


\subsubsection{Gateway 网关暴露和开放 Gateway 网关配置错误}


\textbf{声明：} 在没有认证的情况下绑定到 loopback 之外可能使远程入侵成为可能 / 增加暴露；令牌/密码可以阻止未认证的攻击者（根据模型假设）。

\begin{itemize}
  \item 绿色运行：
  \item \texttt{make gateway-exposure-v2}
  \item \texttt{make gateway-exposure-v2-protected}
  \item 红色（预期）：
  \item \texttt{make gateway-exposure-v2-negative}
\end{itemize}


另见：模型仓库中的 \texttt{docs/gateway-exposure-matrix.md}。

\subsection{v1++：额外的有界模型（并发、重试、追踪正确性）}


这些是后续模型，围绕真实世界的故障模式（非原子更新、重试和消息扇出）提高保真度。

\subsubsection{配对存储并发 / 幂等性}


\textbf{声明：} 配对存储应该在交错情况下也强制执行 \texttt{MaxPending} 和幂等性（即"检查然后写入"必须是原子/加锁的；刷新不应创建重复项）。

这意味着：

\begin{itemize}
  \item 在并发请求下，你不能超过渠道的 \texttt{MaxPending}。
  \item 对同一 \texttt{(channel, sender)} 的重复请求/刷新不应创建重复的活跃待处理行。
\end{itemize}


\begin{itemize}
  \item 绿色运行：
  \item \texttt{make pairing-race}（原子/加锁的上限检查）
  \item \texttt{make pairing-idempotency}
  \item \texttt{make pairing-refresh}
  \item \texttt{make pairing-refresh-race}
  \item 红色（预期）：
  \item \texttt{make pairing-race-negative}（非原子 begin/commit 上限竞争）
  \item \texttt{make pairing-idempotency-negative}
  \item \texttt{make pairing-refresh-negative}
  \item \texttt{make pairing-refresh-race-negative}
\end{itemize}


\subsubsection{入站追踪关联 / 幂等性}


\textbf{声明：} 摄入应在扇出过程中保持追踪关联，并在提供商重试下保持幂等。

这意味着：

\begin{itemize}
  \item 当一个外部事件变成多个内部消息时，每个部分都保持相同的追踪/事件标识。
  \item 重试不会导致重复处理。
  \item 如果提供商事件 ID 缺失，去重回退到安全键（例如，追踪 ID）以避免丢弃不同的事件。
\end{itemize}


\begin{itemize}
  \item 绿色：
  \item \texttt{make ingress-trace}
  \item \texttt{make ingress-trace2}
  \item \texttt{make ingress-idempotency}
  \item \texttt{make ingress-dedupe-fallback}
  \item 红色（预期）：
  \item \texttt{make ingress-trace-negative}
  \item \texttt{make ingress-trace2-negative}
  \item \texttt{make ingress-idempotency-negative}
  \item \texttt{make ingress-dedupe-fallback-negative}
\end{itemize}


\subsubsection{路由 dmScope 优先级 + identityLinks}


\textbf{声明：} 路由必须默认保持私信会话隔离，只有在明确配置时才折叠会话（渠道优先级 + 身份链接）。

这意味着：

\begin{itemize}
  \item 渠道特定的 dmScope 覆盖必须优先于全局默认值。
  \item identityLinks 应该只在明确链接的组内折叠，而不是跨不相关的对等方。
\end{itemize}


\begin{itemize}
  \item 绿色：
  \item \texttt{make routing-precedence}
  \item \texttt{make routing-identitylinks}
  \item 红色（预期）：
  \item \texttt{make routing-precedence-negative}
  \item \texttt{make routing-identitylinks-negative}
\end{itemize}
